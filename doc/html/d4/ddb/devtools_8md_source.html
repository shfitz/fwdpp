3 ## Intro
4 
5 This document covers how to use GNU autotools to manage a project based on [fwdpp](http://molpopgen.github.io/fwdpp/).

7 ## What you need
8 
9 You'll need the following installed on your system
10 
11 * [autoconf](https://www.gnu.org/software/automake/)
12 * [automake](https://www.gnu.org/software/automake/)
13 
14 For Linux users, these are probably already installed on your system.

16 ## Getting the right tools on OS X
17 
18 OS X users are at a slight disadvantage here, as the Xcode tool suite no longer includes the GNU tools setup. However, you can use [homebrew](http://brew.sh) to get these tools. Please see that website for complete instructions, but at the time of this writing, the following will work:
19 
20 1. Make sure that you have Xcode installed. You cannot in stall brew without it.
21 2. Install brew:
22 
23 ~~~{.sh}
24 ruby -e "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/master/install)"
25 ~~~
26 
27 3. Install several packages that you'll need:
28 
29 ~~~{.sh}
30 brew install autoconf automake libtool wget
31 ~~~
32 
33 4. Optionally, install some other handy tools:
34 
35 ~~~{.sh}
36 brew install doxygen
37 ~~~

39 ## Generating a skeleton project
40 
41 Move into the devtools directory of the fwdpp source code repository:
42 
43 ~~~{.sh}
44 cd devtools
45 ~~~
46 
47 There, you'll find an executable bash-language script called setup.sh. It's usage is:
48 
49 ~~~{.sh}
50 Usage: ./setup.sh options
51 Mandatory options are:
52  -d/--dir root path to place the project (DIR)
53  -p/--project Name for project directory (PROJECT)
54 Together, the -p and -d options will create a skeleton package in DIR/PROJECT.
55 For example: ./setup.sh -p ~/src -d my_fwdpp_project
56 Optional options are:
57  -u/--url project url. This must be single-quoted with special characters escaped, e.g. -u 'https:\/\/github.com\/molpopgen\/fwdpp'
58 ~~~
59 
60 Here is an example of using this script to create a project called fwdpp_project in my user's src subdirectory. I intend for the project to be hosted on [GitHub](http://github.com), so I'll pass it an appropriate URL, too:
61 
62 ~~~{.sh}
63 ./setup.sh -d ~/src -p fwdpp_project -u 'https:\/\/github.com\/molpopgen\/fwdpp_project'
64 ~~~
65 
66 The following got printed to the screen:
67 
68 ~~~
69 Base path set to '/Users/krthornt/src'
70 Project name set to 'fwdpp_project'
71 ~~~
72 
73 We're almost done, but our build system isn't 100 percent ready for our system. To finish the job, we need to say:
74 
75 ~~~{.sh}
76 cd ~/src/fwdpp_project
77 autoreconf -fi
78 autoheader
79 automake --add-missing -c
80 ~~~
81 
82 Hint: at this point, it may be prudent to make an initial commit to your version control system!

84 ## Some details
85 
86 On all systems that I work on, the build system now works after the commands in the previous section are completed.
87 
88 Our configure.ac contains the correct project name, URL, etc:
89 
90 ~~~
91 head -n 10 configure.ac
92 AC_PREREQ(2.59)
93 
94 AC_INIT([fwdpp_project],[0.1.0],[https://github.com/molpopgen/fwdpp_project])
95 AC_CONFIG_SRCDIR([src/fwdpp_project.cc])
96 AM_INIT_AUTOMAKE
97 AC_CONFIG_HEADERS([config.h])
98 
99 AC_CONFIG_MACRO_DIR([m4])
100 
101 AC_PROG_CC
102 ~~~
103 
104 Further, src/Makefile.am looks good:
105 
106 ~~~
107 bin_PROGRAMS=fwdpp_project
108 
109 fwdpp_project_SOURCES=fwdpp_project.cc
110 
111 AM_CPPFLAGS=-W -Wall -I. -I..
112 
113 if DEBUG
114 else !DEBUG
115 AM_CPPFLAGS+=-DNDEBUG
116 endif
117 ~~~
118 
119 The configure script checks for the presence of several [boost](http://www.boost.org) libraries, [fwdpp](http://molpopgen.github.io/fwdpp/), [libsequence](http://molpopgen.github.io/libsequence), the [GSL](http://www.gnu.org/software/gsl/), and [zlib](http://zlib.net). It also checks that your compiler is C++11-ready.
